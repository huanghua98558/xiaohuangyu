## 二、任务曝光系统分析

### 2.1 系统概述

任务曝光系统控制任务对用户的可见性，核心目标是：
1. 公平分配任务曝光机会
2. 根据用户等级/活跃度优化曝光
3. 防止恶意刷任务

### 2.2 核心组件

| 文件 | 功能 |
|-----|------|
| exposureService.js | 曝光逻辑核心服务 |
| exposureCron.js | 定时任务调度 |
| exposureQuotaService.js | 用户曝光额度管理 |
| exposureController.js | 控制器 |
| exposureRoutes.js | API路由 |

### 2.3 曝光模式

系统支持三种曝光模式：

1. **priority (优先级模式)** - 当前使用
   - 白名单用户优先级 +100
   - 黑名单用户优先级 -50
   - 综合活跃度、提交速度、完成率计算优先级

2. **parallel (并行模式)**
   - 所有用户同时看到任务

3. **sequential (顺序模式)**
   - 按完成率阈值控制曝光顺序

### 2.4 定时任务

| 任务 | 间隔 | 功能 |
|-----|------|------|
| 曝光检查 | 5分钟 | 检查活跃任务曝光状态 |
| 离线缓冲检查 | 1分钟 | 处理离线用户的曝光缓冲 |
| 质量评分计算 | 10分钟 | 更新用户质量评分 |
| 过期任务释放 | 30秒 | 释放过期任务 |
| 任务池预热 | 2分钟 | 预热任务缓存 |

### 2.5 问题与风险

1. **数据库表缺失**: `exposure_config` 表不存在，使用默认配置
2. **Redis连接问题**: 部分服务未正确使用 getRedisClient()
3. **多数据源混乱**: 同时使用 Supabase 和 Prisma

## 三、链接审查系统分析

### 3.1 系统概述

链接审查系统负责验证用户提交的任务链接，每天处理 30,000+ 条链接。

### 3.2 架构设计

```
┌─────────────────┐
│  BullMQ Queue   │  ← 任务队列
│ link_verify_queue│
└────────┬────────┘
         │
    ┌────┴────┐
    │ Scheduler│  ← 智能调度器 (1实例)
    │  调度任务  │
    └────┬────┘
         │
    ┌────┴────────────────┐
    │         │           │
    ▼         ▼           ▼
┌───────┐ ┌───────┐ ┌───────┐
│Worker1│ │Worker2│ │Worker3│  ← 3个Worker并行处理
└───┬───┘ └───┬───┘ └───┬───┘
    │         │           │
    └────┬────┴───────────┘
         │
    ┌────┴────────────────┐
    │                     │
    ▼                     ▼
┌──────────┐      ┌──────────┐
│ 直连访问  │      │ 代理IP池 │
│ (优先)   │      │ (降级)   │
└──────────┘      └──────────┘
         │              │
         └──────┬───────┘
                │
                ▼
         ┌─────────────┐
         │Browser Service│  ← 3个实例 (8000-8002)
         │ Puppeteer    │
         └─────────────┘
```

### 3.3 核心流程

1. **任务入队**: 用户提交任务 → 创建验证任务
2. **智能调度**: Scheduler 批量调度，避免空跑
3. **多IP重试**:
   - 先尝试直连
   - 失败后获取代理IP
   - 最多尝试3个不同IP
4. **结果处理**: 验证结果写库 + WebSocket通知

### 3.4 性能指标

- **超时配置**: 120秒/链接
- **批量配置**: 
  - 最大批量: 50
  - 时间触发: 2分钟
  - 最小批量: 5
- **并发能力**: 3 Worker × 1 任务 ≈ 3 并发

### 3.5 问题与风险

1. **代理IP不稳定**: 无自有IP池，依赖第三方
2. **浏览器服务内存泄漏**: 长期运行可能内存不足
3. **并发瓶颈**: 仅3并发，日30000链接需优化

## 四、图片审查系统分析

### 4.1 系统概述

图片审查系统负责验证用户上传的截图，每天处理 60,000+ 张图片。

### 4.2 架构设计

```
┌─────────────────┐
│  BullMQ Queue   │  ← 任务队列
│image_review_queue│
└────────┬────────┘
         │
    ┌────┴────────────────┐
    │         │           │
    ▼         ▼           ▼
┌───────┐ ┌───────┐ ┌───────┐
│Worker1│ │Worker2│ │Worker3│  ← 3个Worker并行处理
└───┬───┘ └───┬───┘ └───┬───┘
    │         │           │
    └────┬────┴───────────┘
         │
    ┌────┴────────────────┐
    │                     │
    ▼                     ▼
┌──────────┐      ┌──────────┐
│OCR Service│     │ YOLO Service│
│ x2实例    │     │ x1实例     │
│ RapidOCR │     │ YOLOv8n    │
└──────────┘      └──────────┘
         │              │
         └──────┬───────┘
                │
                ▼
         ┌─────────────┐
         │  百炼AI降级  │  ← OCR/YOLO都失败时
         └─────────────┘
```

### 4.3 审查流程

1. **图片获取**: 从URL下载或本地路径读取
2. **OCR识别**: 提取文字内容
3. **评论匹配**: 检查是否包含目标评论
4. **正面判断**: 检查是否正面评论
5. **YOLO检测**: 检测图片内容
6. **降级处理**: 主服务失败时使用百炼AI

### 4.4 性能指标

- **OCR超时**: 10秒
- **YOLO超时**: 5秒
- **百炼AI超时**: 30秒
- **健康检查**: 30秒间隔
- **失败阈值**: 连续3次标记不健康

### 4.5 问题与风险

1. **YOLO模型过小**: yolov8n 检测精度有限
2. **百炼AI成本**: 降级时增加API成本
3. **图片存储**: 本地存储可能爆满

## 五、安全机制分析

### 5.1 SSL/TLS配置

**当前状态**: ✅ 正常

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:...;
ssl_session_timeout 1d;
ssl_session_cache shared:SSL:50m;
```

**安全头配置**:
- X-Frame-Options: SAMEORIGIN
- X-Content-Type-Options: nosniff
- X-XSS-Protection: 1; mode=block
- Strict-Transport-Security: max-age=31536000

**证书**: Let's Encrypt (配置中指定，需确认)

### 5.2 认证机制

- JWT Token 认证
- 角色权限控制 (admin/reviewer/part_timer/client)
- Token 刷新机制

### 5.3 限流策略

| 类型 | 窗口 | 上限 | 适用场景 |
|-----|------|------|---------|
| API限流 | 15分钟 | 1000次/IP | 通用API |
| 认证限流 | 15分钟 | 100次/IP | 登录注册 |

### 5.4 安全风险

1. **CORS配置**: 允许 localhost，生产环境应移除
2. **无CSRF保护**: 未使用 CSRF Token
3. **密码策略**: 未强制复杂度
4. **日志敏感信息**: 需审计日志脱敏

## 六、优化方案

### 6.1 高优先级 (P0)

#### 6.1.1 统一数据源
**问题**: 同时使用 Supabase 和 Prisma 导致数据不一致  
**方案**: 全面迁移到 Prisma + CockroachDB  
**影响**: 避免用户ID匹配错误等问题

#### 6.1.2 Redis连接优化
**问题**: 部分服务直接导入 redisClient (null)  
**方案**: 统一使用 getRedisClient() 获取连接  
**已完成**: onlineUserService.js 已修复

#### 6.1.3 曝光配置表创建
**问题**: exposure_config 表不存在  
**方案**: 执行数据库迁移创建表  
**SQL**: 
```sql
CREATE TABLE exposure_config (
  id SERIAL PRIMARY KEY,
  is_active BOOLEAN DEFAULT true,
  exposure_mode VARCHAR(50) DEFAULT 'priority',
  -- 其他字段...
);
```

### 6.2 中优先级 (P1)

#### 6.2.1 链接审查并发优化
**问题**: 3并发不足以支撑日30000链接  
**方案**: 
- 增加 Worker 实例到 5-10 个
- 优化 Browser Service 性能
- 建立自有代理IP池

#### 6.2.2 图片审查性能优化
**问题**: OCR/YOLO服务单点  
**方案**:
- YOLO 增加到 2 实例
- 使用更精确模型 (yolov8s)
- 图片预处理压缩

#### 6.2.3 数据库索引优化
**方案**: 为高频查询添加索引
```sql
CREATE INDEX idx_tasks_status_created ON tasks(status, created_at);
CREATE INDEX idx_claims_user_status ON claims(user_id, status);
```

### 6.3 低优先级 (P2)

#### 6.3.1 日志管理
**问题**: 日志可能无限增长  
**方案**: 
- 配置 logrotate
- 日志分级存储
- 敏感信息脱敏

#### 6.3.2 监控告警
**方案**: 
- 部署 Prometheus + Grafana
- 配置服务健康告警
- 错误率监控

#### 6.3.3 自动化部署
**方案**: 
- 配置 CI/CD 流水线
- 蓝绿部署
- 自动回滚
